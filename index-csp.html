CSP test suite

Static pages for Content-Security-Policy experiments. On GitHub Pages, policy is delivered via <meta http-equiv="Content-Security-Policy"> (no server headers). Use separate pages so each load gets one fixed policy — that matches how you will validate on mobile and desktop browsers.

Tip: Prefer relative URLs for assets (./script.js) so the same build works locally and under /repo-name/ on GitHub Pages.

How to test multiple CSP cases
  • One URL per policy. Each HTML file ships a single <meta http-equiv="Content-Security-Policy">. The browser applies it on load; you cannot switch policies without loading another page (or an iframe pointing at another file).
  • Use separate tabs or devices. Open csp/nonce-test.html, csp/case-external-only.html, csp/case-baseline.html in different tabs and compare console / network behavior.
  • Hard refresh (Cmd+Shift+R / Ctrl+Shift+R) after you change HTML so cached documents do not hide updates.
  • DevTools: Console shows “Refused to execute inline script…” when nonce/hash/hosts are wrong. Network shows blocked requests for connect-src / script-src. If the SDK calls wa.appsflyer.com (e.g. /coverdomain), that host must be listed under connect-src (wildcards are not always enough across browsers).
  • Mobile: Safari Web Inspector (macOS) or Chrome remote debugging (USB) — load the same GitHub Pages URLs as on desktop.
  • Production vs static demo: Real sites usually send CSP as an HTTP header with a fresh nonce per response. These pages use a fixed nonce only so GitHub Pages can demonstrate the idea.
Test cases
  • Nonce test (recommended) — static nonce, inline checks, AppsFlyer loader via onload.
  • Nonce + host allowlist — same policy as nonce test, shorter page (legacy).
  • External scripts only — no inline JS; bootstrap lives in csp/af-bootstrap.js (strictest practical for static sites).
  • Baseline (permissive) — includes 'unsafe-inline' for scripts to compare behavior.
  • Main index (no CSP meta) — default integration reference.
